# RULES — エージェントが絶対にやってはいけないこと

> VERIFY の品質ゲートと並ぶガードレール。PLAN / EXECUTE でこのルールを最優先で守り、抵触する
> 場合は変更しない。

## 禁止事項（Hard rules）

- 保護ブランチ（`main`/`master`）へ直接コミット・push しない。作業は必ず作業ブランチで行う
  （現在のブランチがそれか DISCOVER で確認。違えば何も変更せず Open に記録して終了）。
- **`src/` の挙動・公開 API のシグネチャを変えない。** このループの目的は「テスト追加」であり、
  プロダクションコードの振る舞いを変えてはならない（リグレッション禁止）。
  - 例外: テスタビリティ確保のための**最小**リファクタ（例: 純粋関数の export 追加）は可。ただし
    既存の公開挙動・シグネチャ・出力を完全に保つこと。判断に迷えば変更せず Open に記録。
- **カバレッジ水増しの空テストを書かない。** 各テストは実際に挙動を検証する assert を持つこと。
  到達するだけで何も検証しないテスト、自明に真な assert、`expect(true).toBe(true)` 等は不可。
- `vitest.config.ts` の coverage thresholds（95/85）を**下げない・無効化しない**。これは DoD の数値。
- 既存テストを「通すために」削除・スキップ（`.skip` / `.todo`）・無効化しない。
- VERIFY が通っていない変更をコミットしない。
- `docs/`・`manifest.json`・`styles.css`・`newtab.html`・UI コンポーネントの挙動を変更しない
  （テストの追加対象として読むのは可。挙動は変えない）。
- 1周で多数ファイルにまたがる大規模変更をしない（1周 = 1テストファイル追加 or 1モジュールの
  カバレッジ向上、最小変更）。
- `git commit --amend` や同一タスクの追加コミットをしない（1タスク=1コミット）。
- 依存パッケージの追加・更新をしない。
- 秘密情報・認証情報をコード・ログ・コミットに書かない。
- `loop/` 配下（ループ設定）を勝手に書き換えない（`loop/MEMORY.md` への追記は除く。MEMORY は
  毎周更新する）。

## 進め方の原則

- これは自律実行ループ。`brainstorming`/`writing-plans` 等のメタ/プロセス系スキルは起動せず、
  `LOOP_PROMPT.md` の5段階を直接実行する。
- 判断に迷ったら**安全側（変更しない）**に倒し、`loop/MEMORY.md` の Open に判断ポイントを記録する。
- VISION.md のベースラインで挙げた低カバレッジ・ファイルから優先的に攻める（伸びしろが大きい）。
- happy-dom や Chrome API モックで再現困難なパスは無理に水増しせず、Open にその旨を残す。
- 「テストが緑」≠「完了」。完了ゲート（coverage しきい値 + build:extension）が全緑になるまで
  `LOOP_DONE` を出さない。
